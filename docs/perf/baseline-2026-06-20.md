## Known Pre-Existing Render Condition (NOT caused by this plan)

The page does not produce a contentful paint because the current data snapshot does not match
what `index.html` expects. The bootstrap guard at `index.html` validates:

```js
const DATA = window.SUPPLY_MAP_DATA;
if (!DATA || !Array.isArray(DATA.nodes) || !Array.isArray(DATA.links) || !DATA.profiles) {
  showFatalError("Missing or invalid data/top100-map.js");
  throw new Error("Missing or invalid data/top100-map.js");
}
```

The committed `data/top100-map.js` snapshot sets `window.SUPPLY_MAP_DATA` with keys
`last_auto_update, update_source, snapshot_date, top100_source_url, update_policy, companies,
change_log` — but **no `nodes`, `links`, or `profiles`** arrays. The guard therefore throws
(`pageerror: Missing or invalid data/top100-map.js`) and the D3 visualization never renders,
so Chrome never records an FCP/LCP. Additionally, the sandbox blocks the Google Fonts CDN
(`net::ERR_BLOCKED_BY_ORB`), which would also suppress font rendering.

This is the **pre-existing runtime state of the repo** (a stale/format-mismatched data snapshot
plus a network-restricted environment) — it is NOT introduced by Phase 01 extraction. This plan
(01-01) writes only under `docs/perf/` and changes no source file. The condition is logged here
so the post-extraction comparison is made under identical conditions (same data snapshot, same
environment), and so plans 01-02/01-03 are aware that visual render verification on this machine
requires either a data snapshot containing `nodes`/`links`/`profiles` or running outside the
CDN-blocking sandbox.

## Module-MIME Serving Confirmation (resolves research assumption A1)

`npx http-server` was smoke-tested:

| Path | Status | Content-Type |
|------|--------|--------------|
| `/index.html` | 200 | `text/html; charset=UTF-8` |
| `/data/top100-map.js` | 200 | `application/javascript; charset=UTF-8` |
| `/data/credit-ratings.js` | 200 | `application/javascript; charset=UTF-8` |

`http-server` serves `.js` as `application/javascript`, which is a valid JavaScript MIME type
for `<script type="module">` per the WHATWG spec (both `text/javascript` and
`application/javascript` are accepted module MIMEs). **No `--mimetypes` override is required**;
local ES-module serving for plans 01-02/01-03 is confirmed working. (A1 resolved.)
